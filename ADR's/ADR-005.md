## ADR-005 — Contenedorización del Entorno de Base de Datos con Docker Compose

**Título:** Definir una estrategia de despliegue local reproducible usando Docker Compose para PostgreSQL y Liquibase.

**Contexto:**
El proyecto debe poder ejecutarse en cualquier equipo de desarrollo sin depender de instalaciones locales de PostgreSQL ni de configuraciones manuales. El instrumento de evaluación exige que tanto PostgreSQL como Liquibase puedan levantarse mediante contenedor. El equipo trabaja en entornos heterogéneos (Windows, macOS, Linux) y se necesita una forma estándar de inicializar el entorno de base de datos.

**Problema:**
Sin un entorno contenedorizado, cada desarrollador configura su PostgreSQL local de forma diferente, con versiones distintas, collations distintas y sin garantía de que el estado inicial sea idéntico. Esto genera errores difíciles de reproducir y hace que la experiencia de onboarding sea costosa. Además, Liquibase necesita conectarse a PostgreSQL en un orden definido, lo que requiere orquestación.

**Decisión:**
Se definirá un `docker-compose.yml` en la carpeta `docker/` con los siguientes servicios:

yaml

```yaml
services:
postgres:
image: postgres:16-alpine
environment:
      POSTGRES_DB, POSTGRES_USER, POSTGRES_PASSWORD (vía .env)
volumes:
- pgdata:/var/lib/postgresql/data
- ../scripts/roles_setup.sql:/docker-entrypoint-initdb.d/01_roles.sql
ports:
-"5432:5432"
healthcheck: pg_isready

liquibase:
image: liquibase/liquibase:4.x
depends_on:
postgres:
condition: service_healthy
volumes:
- ../liquibase:/liquibase/changelog
command:--url=...--changeLogFile=changelog-master.xml update
```

Las credenciales se gestionan mediante un archivo `.env` que nunca se versiona (`.gitignore`). El volumen `pgdata` persiste los datos entre reinicios.

**Justificación técnica:**
Docker Compose garantiza reproducibilidad total del entorno: misma versión de PostgreSQL, misma configuración de roles, mismas migraciones aplicadas en el mismo orden. La condición `service_healthy` en el servicio de Liquibase evita fallos por race condition al iniciar. Al separar la inicialización de roles (`roles_setup.sql`) de los changelogs de Liquibase, se mantiene el principio de responsabilidad única: Docker configura el motor, Liquibase gestiona el esquema. Este enfoque es directamente portable al pipeline CI/CD en una fase posterior.

**Consecuencias / Impacto esperado:**

* Cualquier aprendiz o colaborador puede levantar el entorno completo con `docker compose up` en menos de un minuto.
* El contenedor de Liquibase aplica automáticamente todos los changelogs pendientes al iniciar, sin intervención manual.
* El archivo `.env.example` se versiona como referencia, pero `.env` permanece fuera del repositorio.
* Este ADR es el fundamento técnico de HU-003 y HU-004 del backlog, y su implementación puede iniciarse como primera HU priorizada durante la prueba supervisada.
