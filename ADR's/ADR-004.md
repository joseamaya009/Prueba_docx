## ADR-004 — Estrategia de Versionamiento del Repositorio con Ramas develop / qa / main

**Título:** Definir un flujo de trabajo Git con tres ramas permanentes para controlar la evolución del proyecto en distintos entornos.

**Contexto:**
El proyecto está en fase de estructuración inicial. No existe aún un repositorio con historia de commits. Toda la base de trabajo es el archivo `modelo_postgresql.sql` y los documentos de la prueba supervisada. Desde el inicio se necesita una estrategia que permita trabajar en paralelo sobre el modelo, validar cambios antes de promoverlos y mantener un estado estable en producción.

**Problema:**
Trabajar directamente sobre una sola rama (`main` o `master`) impide aislar trabajo en progreso, hace imposible implementar revisión por pares antes de aplicar cambios, y no permite diferenciar entre lo que está "en desarrollo", "en pruebas" y "listo para producción". En un proyecto de base de datos, un cambio DDL mal aplicado puede ser destructivo e irreversible sin rollback.

**Decisión:**
Se adoptará el siguiente modelo de ramas permanentes:

```
main        ← estado productivo, solo recibe merges desde qa con aprobación
  ↑
qa          ← entorno de validación, recibe merges desde develop tras revisión
  ↑
develop     ← rama de integración, recibe feature branches de trabajo diario
```

Las ramas de trabajo (`feature/`, `fix/`, `chore/`) se crean desde `develop` y se integran de vuelta a `develop` mediante pull request. Ningún commit directo está permitido en `qa` ni en `main`. Los changelogs de Liquibase se desarrollan en `feature/` branches y se integran al flujo con sus changesets numerados.

**Justificación técnica:**
Este modelo es una simplificación de Gitflow adaptada a proyectos de base de datos con ciclos de validación intermedios. La rama `qa` funciona como espejo del entorno de pruebas donde Liquibase aplica las migraciones contra una base limpia antes de aprobar la promoción a `main`. El uso de pull requests como mecanismo de integración garantiza revisión técnica y registro de la decisión. La separación develop/qa/main es coherente con los tres entornos que levantará Docker Compose (local, test, prod-like).

**Consecuencias / Impacto esperado:**

* Nunca habrá DDL no revisado en el entorno de producción.
* Los ADR, changelogs y scripts de datos de prueba evolucionan bajo el mismo control de versiones que el código.
* El `README.md` del repositorio documentará el flujo de integración para que cualquier colaborador nuevo lo entienda sin ambigüedad.
* La estructura de carpetas del repositorio (`docs/`, `adr/`, `docker/`, `liquibase/`) se crea en el primer commit de `develop` como parte de HU-002.
