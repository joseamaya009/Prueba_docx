## ADR-003 — Implementación de Liquibase para Versionamiento Controlado del DDL

**Título:** Adoptar Liquibase como herramienta oficial de versionamiento, aplicación y trazabilidad del DDL del proyecto.

**Contexto:**
El modelo actual existe como un único archivo `modelo_postgresql.sql` sin historial de cambios, sin numeración de versiones y sin mecanismo que garantice que el estado de la base de datos en cada entorno (local, qa, producción) corresponde exactamente a una versión conocida. Cualquier cambio estructural hoy requiere aplicarse manualmente y confiar en la memoria del equipo.

**Problema:**
Sin control de versiones del DDL, es imposible garantizar paridad entre entornos, reproducir el estado exacto de la base en una fecha determinada, revertir un cambio fallido de forma controlada, ni auditar quién aplicó qué cambio y cuándo. Esto es crítico en un sistema con dominios como pagos, facturación e identidad.

**Decisión:**
Se implementará Liquibase con la siguiente organización:

```
liquibase/
├── changelog-master.xml          ← punto de entrada, incluye todos los dominios
└── changelogs/
    ├── changelog-geography.xml
    ├── changelog-airline.xml
    ├── changelog-identity.xml
    ├── changelog-security.xml
    ├── changelog-customer.xml
    ├── changelog-airport.xml
    ├── changelog-aircraft.xml
    ├── changelog-flight-ops.xml
    ├── changelog-sales.xml
    ├── changelog-boarding.xml
    ├── changelog-payment.xml
    ├── changelog-billing.xml
    └── changelog-crew.xml        ← nuevo dominio ADR-001
```

Cada `changeset` dentro de un changelog tendrá autor, id único y comentario descriptivo. El `changelog-master.xml` incluye los archivos de dominio en orden de dependencia.

**Justificación técnica:**
Liquibase es agnóstico al motor de base de datos, soporta XML, YAML, JSON y SQL como formato de changelog, y mantiene las tablas `DATABASECHANGELOG` y `DATABASECHANGELOGLOCK` que registran el historial completo de aplicación. Al organizar por dominio funcional (máximo un changelog por dominio, según el instrumento) se garantiza que los cambios dentro de un dominio son cohesivos, trazables y fácilmente revisables en pull requests. Liquibase puede ejecutarse desde contenedor en el pipeline o manualmente en local.

**Consecuencias / Impacto esperado:**

* Todo cambio estructural futuro se realiza exclusivamente a través de un nuevo `changeset`, nunca editando un changeset existente.
* Los entornos `develop`, `qa` y `main` aplican automáticamente los changelogs pendientes al iniciar el contenedor.
* Liquibase se convierte en la única fuente de verdad del estado del DDL, reemplazando el archivo monolítico.
* Se requiere que el rol `role_liquibase` tenga permisos suficientes para crear y modificar esquemas (coherente con ADR-002).
